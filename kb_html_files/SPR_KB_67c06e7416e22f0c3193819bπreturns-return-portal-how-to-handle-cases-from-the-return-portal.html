Versions of the Return Portal

We are rolling out the Return Portal across different markets. While we are in the process of integrating with more carriers in all markets, we maintain different versions of the return portal depending on the country. Please check carefully which option applies to your market:

  • Version 1: No automation, for any product.

    • Italy (IT)

    • Ireland (IE)

  • Version 2: Accessory returns are fully automated, 1 or 2 men handling returns need to be fully processed by the agents.

    • Austria (AT)

    • Belgium (BE)

    • Switzerland (CH)

    • Germany (DE)

    • Netherlands (NL)

    • United Kingdom (UK)

  • Version 3: Accessory returns are fully automated, 1 or 2 man handling returns are semi-automated with integrations with carriers.

    • France (FR,

    • Portugal (PT)

    • Spain (ES)

For detailed instructions, jump directly to the version appliccable to your country.

Version 1: Agent Instruction

Applies to Fluent Commerce returns in Italy and Ireland.

First, the request comes from the Customer's side:

  1. Customer logs into the return portal, using order number and email address

  2. Customer selects the products to return/claim

  3. Customer selects the return reason, condition (unpacked/packed) as well as product quantity to return

  4. Customer verifies their contact information

  5. Customer submits return request

After the case is created, Agent's instructions are:

  1. Check the automated email / case: Submitted return request will create an automated email to customer service, that provides Agent with a structured overview of the return request and all required information for the next steps

  2. Validate request - Trial Period:

    ​

    If in trial period: the agent should first attempt deflection. Proceed to step 3: Deflection

    Italy: Until further notice, do not move to step 3. Move directly to step 4.

    If not in trial period:

    The case becomes Product Claim case instead of a Return. Proceed as a Product claim.

  3. Deflection: If the request is validated, you will first attempt deflection, according to your Market.

    ​

    ​Italy: Until further notice, do not move to step 3. Move directly to step 4.

    ​

    1. Verbal deflection (no supplements): Follow the Verbal Deflection KB:

      Return Deflection: Making Impact on Returns

    Note: If deflection is sucessful (either via step 1 or step 2), because the customer accepted to keep their product(s), do NOT create a return on FluentCommerce.

    ​

    1. Verbal deflection (no supplements): Follow the Verbal Deflection KB:

      Return Deflection: Making Impact on Returns

      Note: Only move to step 2 If step 1 is NOT sucessful

      Suplement a Comfort Layer, on the original order, on FluentCommerce if:

Version 2: Agent Instructions

Applies to Fluent Commerce returns in Austria, Belgium, Switzerland, Germany, Netherlands and UK.

For accessory only returns, customers must start the returns by themselves. Every request opened by the customer on the portal, will automatically create a return request and generate a label for the drop-off. No action will be required from the CS agents. The integration between Return Portal, FluentCommerce, and ParcelLab will handle the return automatically.

If the customer reaches out to customer support for returning a standalone accessory, you have two options:

  1. Direct the customer to the Return Portal and explain them how to use it.

    OR:

  2. Help them by creating the return via the Return Portal on their behalf.

*Products considered as accessories for return via drop-off: pillows, bed linens, duvets, mattresses protectors, disassembled/self-assembly slatted frames, baby/kids mattresses, packed toppers or toppers up to 120cm.

**Exclusively accessories refers to returns where exclusively accessories are being returned. No other product type are part of the created return.

If the return request opened by the customer is for a mattress or bed, the customer request will be sent for deflection. This will happen with the portal creating a Sprinklr ticket that needs follow-up actions from CS/Sleep Experts.

All cases coming from the Return Portal will always appear with "from: no-reply@emma-sleep.com". However, it doesn't mean they are related to actual customer cases. An automation assigns them to the right group according to the market from where the order is from.

Step-by-step process

Step 1: Identify the ticket from the Return Portal.

Step 2: Double-check if a request is already open in FluentCommerce to avoid issues with ParcelLab automations.

If a return is associated with a carrier on ParcelLab, do not create a return on FluentCommerce. [Link to ParcelLab]

  • Situation 1: If the ticket is found, assign it to yourself and proceed with creating the return. Inform the customer if the ticket is already assigned.

  • Situation 2: If the ticket is not found, do not create a return on OMS. Inform the customer that everything is as expected and check ParcelLab for email confirmation.

Step 3: Confirm the return in ParcelLab, then check the ticket for "Reason for return" and "Compensation Method" in Freshdesk/Sprinklr.

  • Option 1 (Refund): Customer wants a refund. Try to deflect the return. If not possible, contact the customer about the return condition. Create a return with a refund in FluentCommerce, send a new email to the customer, and resolve the ticket in Sprinklr.

  • Option 2 (Replace): Customer wants a replacement. Follow the Logistics or Product Claim processes. Check if the claim is valid and if all images are provided. Create the return in FluentCommerce, send a new email to the customer, and resolve the ticket in Sprinklr.

Double-check if a request is not already open in FluentCommerce to avoid issues like creating a return and blocking automation from ParcelLab.
We have faced cases where customers open the request via the portal and end up also calling the support line. If this happens and the agent also creates a return in FluentCommerce, the requests will collide and probably not work correctly. 

On ParcelLab, if there is a return, the agent can see if it is associated with a carrier or not. And then should never create a return on FluentCommerce in case there is a label of a carrier: 

​

  • Situation 1: In case the agent finds example 1 above, they must search for the ticket on Sprinklr, assign it to themself and proceed with the return approval. If ticket is already assigned, the agent must inform the customer that everything is as expected and an email will go soon;

  • Situation 2: In case agent finds example 2 above, the agent should not create any return on OMS and inform the customer that everything is going as expected and customer will receive an email soon (check ParcelLab to see email send out and confirm if email was already sent).

After confirming in ParcelLab, go back to Sprinklr and check in the ticket the information provided by customers about 

  • "Reason for return", 

  • "Item Condition" and 

  • "Compensation Method". At the "Compensation method" field, customer can choose between 2 options: 

    • Refund:

      It means that the customer is not satisfied with the item/items purchased and wants to return and be refunded. You can still try to deflect the return, and if not possible, just proceed as a normal return process.

      Move to step 5.

    • Replace:

      It indicates an issue, a claim from the customer regarding their item, and that they want to have a new one. Here, the procedure is to follow the Logistics Claim or Product Claim processes, after investigating the customer issue. 

      Move to step 6.

The deflection is to be done after the agents receive the ticket from the Return Portal with the frequent from the customer.
